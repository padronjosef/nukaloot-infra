# Nukaloot Infra

Infrastructure and orchestration for the Nukaloot project. Docker Compose for local dev, Terraform for AWS production.

## Tech Stack

- Docker Compose (dev and prod)
- Terraform (AWS provisioning)
- Nginx (reverse proxy)
- Cloudflare (SSL, DNS, health checks)
- GitHub Actions (CI/CD, native ARM runners)

## Architecture

```
Internet --> Cloudflare (SSL) --> Nginx (port 80) --> Next.js Web --> NestJS API --> PostgreSQL (Docker)
```

- **Cloudflare**: Handles SSL termination, DNS, and health checks.
- **Nginx**: Reverse proxy on port 80. Only accepts traffic from Cloudflare IPs.
- **Next.js (web)**: Serves the frontend. Proxies API calls internally to the backend.
- **NestJS (api)**: Internal only. Only reachable via Docker's internal network.
- **PostgreSQL**: Runs as a Docker container with a persistent volume. Internal only.

No one outside can reach the API or the database directly.

## Getting Started

### Prerequisites

- Docker installed
- All 3 repos cloned as siblings:

```
parent-folder/
  nukaloot-api/
  nukaloot-web/
  nukaloot-infra/    <-- you are here
```

### Development

```bash
docker compose up
```

| Service | Port | Description |
|---------|------|-------------|
| **web** | `3003` | Next.js frontend with hot-reload |
| **api** | `3002` | NestJS backend with hot-reload |
| **db** | `5434` | PostgreSQL 16 |

Source code is mounted as volumes. Changes to `nukaloot-api/src/` and `nukaloot-web/src/` are picked up automatically.

### Environment Variables

`deploy.sh` reads `WEB_APP_DOMAIN` from `.env` for nginx config regeneration.

## AWS Infrastructure (Terraform)

| Resource | Spec |
|----------|------|
| EC2 | t4g.micro (ARM), 15GB gp3 EBS |
| S3 | Terraform state bucket (versioned) |
| IAM | Instance profile with CloudWatchAgentServerPolicy |
| CloudWatch | Agent (disk, memory metrics), 3 alarms (CPU >80%, disk >80%, memory >80%), SNS email alerts, dashboard |
| SNS | Alert topic with email subscription |

No Elastic IP — Cloudflare DNS points to the EC2 public IP. PostgreSQL runs locally in Docker (no RDS). Deploy key is generated by Terraform and passed via user-data (no Secrets Manager). AMI is pinned as a variable in tfvars (gitignored).

### Terraform Setup

```bash
cd terraform
./setup.sh                    # One-time: create S3 state bucket
# Create terraform.tfvars with domain and ami_id
terraform init
terraform apply -var="ssh_public_key_path=/path/to/key.pub"
```

After apply: add the `github_deploy_public_key` output as an SSH key on your GitHub account. Add `EC2_HOST` and `EC2_SSH_KEY` as secrets to all 3 repos for CI/CD.

## Project Structure

```
nukaloot-infra/
  docker-compose.yml          # Local development
  docker-compose.prod.yml     # Production (Nginx + Web + API + PostgreSQL)
  nginx/
    nginx.conf.template       # Nginx template (Cloudflare handles SSL)
    nginx.conf                # Generated (do not edit)
  terraform/
    main.tf                   # AWS provider, S3 backend
    variables.tf              # Input variables
    ec2.tf                    # EC2 instance, key pair, instance profile
    monitoring.tf             # IAM role, CloudWatch alarms, SNS, dashboard
    network.tf                # Security groups (Cloudflare-only ingress)
    outputs.tf                # IP, domain, SSH command, deploy key
    setup.sh                  # One-time S3 bucket creation
    scripts/
      user-data.sh            # EC2 bootstrap (Docker, swap, clone, start)
      deploy.sh               # CI/CD deploy script (prune, pull, restart)
      cloudwatch-agent-config.json  # CloudWatch Agent metrics config
```

## Deployment

All 3 repos deploy on push to `main` via GitHub Actions (native ARM runners `ubuntu-24.04-arm`) → SSH to EC2.

`deploy.sh` prunes old Docker images before pulling new ones, then:

```bash
deploy.sh api    # Pull and restart API container
deploy.sh web    # Regenerate nginx config, pull and restart web + nginx
deploy.sh infra  # git fetch + reset --hard, regenerate nginx, pull and restart all
deploy.sh all    # Same as infra
```

Version bump and GitHub Release are created **after** a successful deploy. Job timeouts: API 8min, web 10min, infra 3min.

## Auto-Versioning

All 3 repos use the same convention: prefix commit messages with `[major]`, `[minor]`, or `[patch]` to auto-bump versions via GitHub Actions.
